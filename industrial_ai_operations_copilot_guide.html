1. Business Context

Click the magnifier or the image to zoom in.

A chemical company with 3 production plants (Valencia, Tarragona, Bilbao) needs a tool that unifies its operations: production, maintenance, alerts, vehicle fleet, and technical support. The goal is to help plant managers, reliability engineers, and maintenance technicians make better decisions, faster, with the help of artificial intelligence.

The Real Problem

  • Equipment, sensor, alert, and maintenance information is scattered across different systems
  • When an alert fires, the technician spends hours searching for the right manual, reviewing the history, and deciding what to do
  • Production rescheduling decisions are made by phone, without real-time data
  • Predictive maintenance exists as a concept but doesn't translate into concrete actions

What We Built

A complete application with 8 pages, realistic synthetic data of 2.8M+ sensor readings across 446 equipment and 1,470 sensors, integrated AI functions, semantic search over technical manuals, and a natural language chat that answers operational questions with real data.

🏭

Operations Center

Benefit: Informed decision in 10 seconds

3. The Prompts: How We Built It

Each step is executed with a natural language prompt inside Cortex Code. No code is written manually. Below is each actual prompt and what it generates in Snowflake.

1 Create the database, structure, and synthetic data
/best-practices Create a demo in Snowflake called INDUSTRIAL_AI_OPERATIONS_COPILOT. I need a database, schemas, and a demo warehouse. 3 plants: A24, B17, C09. 4 buildings per plant. Production zones: Polymerization, Pelletizing, Packaging, Coating, Compression, Purification. Industrial equipment: pumps, reactors, compressors, tanks, conveyor belts, coolers. Sensors for temperature, vibration, pressure, flow, and energy. Synthetic tables for plants, buildings, production zones, production lines, equipment, sensors, sensor readings, alerts, work orders, tickets, manuals, production orders, inventory, maintenance history, fleet vehicles, and customer requests. Time series spanning at least 90 days. Anomaly, degradation, and imminent failure events. Include data patterns that enable alerts for high temperature, performance drop, pump failure risk, inventory shortage, and production rescheduling. Relationships between objects like Plant > Building > Zone > Equipment > Sensor > Alert > Work Order. Analytical views for operational KPIs.
Generated in Snowflake: 1 database (INDUSTRIAL_AI_OPERATIONS_COPILOT), 1 warehouse, 2 schemas (RAW_DATA, ANALYTICS), 16 tables with 2.8M+ synthetic records with embedded real patterns, 8 analytical views. 2,831,760 sensor readings (90-day hourly time series with Normal/Anomaly/Degradation/Failure events) across 1,470 sensors and 446 equipment. 966 alerts with 10 types. 150 tickets, 452 work orders, 72 production orders with rescheduling. 36 inventory items, 2,230 maintenance history records. 14 fleet vehicles, 48 customer requests.
2 Configure semantic search with Cortex Search
/best-practices Configure Cortex Search for technical documents and historical tickets. Create a DEMO_MANUAL_CHUNKS table with synthetic technical documents. A DEMO_TICKET_KNOWLEDGE_BASE table with historical tickets, comments, and resolutions. A Cortex Search service over both content sources.
Generated: 1 Cortex Search service (INDUSTRIAL_OPS_SEARCH) with snowflake-arctic-embed-m-v1.5 model, 35 indexed documents (20 technical manual chunks + 15 historical ticket records) via unified SEARCH_CONTENT view. Functional semantic search with filters by SOURCE_TYPE, EQUIPMENT_TYPE, CONTENT_CATEGORY, TITLE, and SECTION.
3 Create AI functions with Cortex
/best-practices Create SQL functions that use Snowflake Cortex AI to: summarize a technical ticket, classify alert severity, extract equipment, zone, symptom, and probable cause from free text, generate a troubleshooting recommendation, generate an explanation of why an alert should be approved, rejected, or escalated, and generate a production rescheduling proposal.
Generated: 6 SQL functions (UDFs) using SNOWFLAKE.CORTEX.COMPLETE (and AI_CLASSIFY / AI_EXTRACT) with mistral-large2 model and structured prompts returning JSON: FN_SUMMARIZE_TICKET, FN_CLASSIFY_ALERT_SEVERITY (AI_CLASSIFY), FN_EXTRACT_INCIDENT_ENTITIES (AI_EXTRACT), FN_TROUBLESHOOTING_RECOMMENDATION, FN_ALERT_DISPOSITION, FN_PRODUCTION_RESCHEDULE_PROPOSAL.

Generated: Streamlit application structure (streamlit_app.py) with 8 navigation pages via st.navigation/st.Page, custom industrial CSS with Snowflake brand palette and connection to all tables. Pages are implemented in prompts 5 through 12.

What Each Object Type Does

Tables (16)

Store operational data: plants, equipment, sensors, IoT readings, alerts, tickets, production orders, maintenance history, vehicles, and spare parts. They are the platform's "source of truth."

Views (8)

Predefined queries that join data from multiple tables to calculate KPIs: equipment health, plant operational KPIs, sensor anomaly dashboard, maintenance cost analysis, alert response metrics, inventory risk, production performance, and a unified search content view for Cortex Search.

Cortex AI UDFs (6)

SQL functions that call a language model (mistral-large2) with structured prompts. They summarize tickets, classify alerts, extract information from free text, generate troubleshooting plans, and rescheduling proposals.

Cortex Search (1)

Semantic search service (INDUSTRIAL_OPS_SEARCH) with snowflake-arctic-embed-m-v1.5 model over 35 documents via the SEARCH_CONTENT unified view (manuals + historical tickets). Searches by meaning, not just exact keywords. Used in Ticket Assistant, Alerts, Fleet Copilot, and Ask AI.

Streamlit Application (1)

Complete web application with 8 pages, deployed in Snowpark Container Services. Reads data directly from Snowflake, calls AI UDFs, and uses Cortex Search. No external server.

Warehouse + Compute Pool

DEMO_INDUSTRIAL_WH executes SQL queries. STREAMLIT_COPILOT_POOL is the container node where the Streamlit application runs persistently.
